项目概述：
这是一个研究/教学导向的 CPU tensor compiler prototype，稳定主路径是 `Tensor -> TraceContext -> Graph -> optimize_graph -> CppCodegen -> JITEngine/runtime`。

前向路径：
前端提供统一的 `Tensor` API，在 eager 模式下直接做 NumPy 计算；如果开启 `TraceContext`，同一套算子还会被记录成计算图里的节点。
中端会在这张 `Graph` 上做 constant folding、CSE、fusion 和 DCE 等图优化。
后端再把优化后的计算图 lower 成 C++ kernel，并生成 shape、workspace 和输入约束等元数据；运行时负责调用本地编译器生成动态库，最后返回一个 `JITModule` 来执行。

反向路径：
除了前向编译执行，这个项目还实现了两条反向梯度路径。
在 eager 模式下，`Tensor` 会在前向阶段记录前驱节点和局部 backward 逻辑，调用 `backward()` 时按逆拓扑顺序传播梯度。
在 graph-mode 训练路径里，系统维护了一套统一的 VJP registry，每个算子同时定义 eager 版和 graph 版梯度规则；训练阶段会基于 forward graph 自动构造 backward graph，再交给同一套 JIT 后端编译。
也就是说，前向和反向不是完全割裂的两套系统，而是共享同一套梯度数学定义，只是执行形式不同。

项目价值：
这个项目的价值在于，它不是只做到 tracing 或 pass，而是把前向图优化、native codegen、runtime 执行，以及 eager backward、graph-mode backward、训练步骤编译完整打通了。
当前 stable path 仍然是收窄的，比如只支持单输出，默认不支持 control flow lowering，runtime 还是基于 `ctypes`，HPC 2D matmul 需要 exact traced shapes。

压缩版：
这是一个 CPU tensor compiler prototype，稳定主路径是 `Tensor -> TraceContext -> Graph -> optimize_graph -> CppCodegen -> JITEngine/runtime`，前向上支持 eager 和 tracing，并把计算图编译成 C++ kernel 本地执行。
除了前向链路，这个项目还实现了两条反向路径：eager 模式下通过 `Tensor.backward()` 逆拓扑回传梯度，graph-mode 下通过统一的 VJP registry 把 forward graph 构造成 backward graph，再交给同一套 JIT 后端编译。
它的重点是把前向、反向和训练编译的端到端闭环打通了，但 stable path 目前仍限制在单输出、无默认 control flow lowering、`ctypes` runtime 这类较窄边界内。

更口语化版本：
这个项目本质上是在做一个 CPU 上的张量编译器原型。整体上我先用统一的 Tensor API 同时支持 eager 执行和 tracing，把前向计算捕获成一张计算图；然后在图上做一些经典优化，比如常量折叠、公共子表达式消除、融合和死代码消除；最后把图编译成 C++ kernel，用本地 JIT 跑起来。
除了前向，我还做了反向传播这条链路。eager 模式下是按逆拓扑顺序执行局部梯度传播，graph-mode 下则是基于统一的梯度规则把 forward graph 自动构造成 backward graph，再交给同一套后端编译。
所以这个项目比较有价值的地方是，它不是只做到图构建或者代码生成，而是把前向、反向和训练编译的闭环都打通了。



前端先把 Tensor 表达式 trace 成一张 Graph
可选地对这张图做优化，比如 fusion、DCE
CppCodegen 根据这张图生成一段 C++ 源码
JITEngine 在运行时把这段源码写到临时目录
再调用本机的 clang++ / g++ / cl 编译成 .so 或 .dll
然后用 ctypes 把这个动态库加载进 Python 进程
得到一个 JITModule
后续 JITModule.run() 就可以把 NumPy 输入包装成 ABI 描述符，调用这个 native kernel 执行

前向阶段如果张量需要梯度，系统会同时记录前驱节点和局部梯度传播规则
在 eager 模式下，Tensor.backward() 会先对当前计算过程做拓扑排序
然后按逆拓扑顺序执行每个节点的局部 _backward 逻辑，把梯度逐步传回输入张量
在 graph-mode 训练路径里，系统不会直接执行 eager 的 _backward，而是从 forward graph 出发构造 backward graph
统一的 VJP registry 会把每个 forward op 映射成对应的梯度图片段
从 loss 节点开始放一个梯度 seed，也就是标量 loss 对自身的梯度 1
然后按 forward graph 的逆拓扑顺序遍历，把输出梯度分解并累计到各输入节点
对每个目标参数或输入，最终得到一张对应的梯度 graph
这些 backward graph 会再交给同一套 JITEngine 编译成 native kernel
训练阶段运行时，会先执行 forward loss kernel，再执行各参数的 gradient kernel
如果需要参数更新，还会继续执行 SGD 或 Adam 的 update kernel
也就是说，这个项目里反向传播不是单独一套解释执行逻辑，而是既支持 eager 逆拓扑回传，也支持把梯度计算本身构造成图，再交给同一套 JIT 后端编译执行。

反向图不是特殊结构，而是从 loss 开始按逆拓扑顺序，把每个 op 的梯度规则展开成新的基础算子图，比如 mean 会引入 broadcast，bias 的梯度会引入 reduce，matmul 的梯度会引入 transpose 和新的 matmul。
最后这些梯度图会和前向图一样被 JIT 编译成本地 kernel 执行，所以这个项目的特点是前向和反向都能走同一套图编译链。
