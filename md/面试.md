**必须准备的 5 段回答**

1. `2-3` 分钟项目介绍  
你要能顺着这条线讲：
- 这是一个研究/教学导向的 CPU tensor compiler prototype
- stable path 是 `Tensor -> TraceContext -> Graph -> optimize_graph -> CppCodegen -> JITEngine/runtime`
- 支持 eager、trace、graph optimization、native C++ JIT、训练步骤编译
- 目前明确限制是单输出、默认不支持 control flow lowering、runtime 还是 ctypes-based

2. 一个完整例子  
推荐就讲 `relu(x @ w + b)`。  
你要说清：
- `Tensor` 在 trace 模式下追加节点
- `TraceContext` 收集成 `Graph`
- `optimize_graph` 可能把模式融合成 fused op
- `CppCodegen` 把图 lower 到 C++ kernel
- `JITEngine` 本地编译并返回 `JITModule`
- `run()` 做输入校验、分配输出和 workspace，然后执行 native kernel

3. 正确性保护故事  
推荐讲这 3 个：
- rank mismatch 在 Python runtime 先拦截
- HPC 2D matmul 用 exact traced shape guard 避免 silent wrong answer
- benchmark 每个 case 都先和 NumPy reference 对齐，再计时

4. 性能/工程化故事  
推荐讲这 4 个：
- graph fusion
- memory planner / workspace reuse
- HPC matmul specialization
- benchmark baseline + CI smoke

5. 项目短板和路线图  
要坦诚讲：
- 当前 stable path 刻意收窄
- control flow、layered backend、pybind runtime 还在 experimental
- runtime 未来应继续下沉到 C++
- benchmark 现在适合趋势判断和回归检测，不等于完整 HPC 实验体系

**面试时高概率会被问的问题**
把这些题自己提前口述一遍：

- 为什么用 Python 做这个项目的前中端？
- 为什么 IR 现在这么轻？
- pass pipeline 为什么是 `constfold -> cse -> fusion -> dce`？
- 为什么不支持多输出？
- `ctypes` runtime 的优缺点是什么？
- 你怎么验证优化没有破坏正确性？
- 为什么 control flow 还没并入 stable？
- 这个项目和 TVM / XLA / PyTorch Dynamo 的差别是什么？

**你要避免的讲法**
- 不要说“工业级”
- 不要说“已经完整支持 control flow / pybind / layered backend”
- 不要拿它直接对标 TVM/XLA
- 不要把 experimental 讲成 stable
