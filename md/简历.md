## 3. 面试高频问题与参考回答

### Q1：你的框架是动态图还是静态图？
**答**：三种前端都支持。默认是动态图（eager + tracing）；新增了惰性前端（LazyTensor）；也实现了严格静态图前端（StaticGraph，define-and-run）。运行时统一可以编译到 C++ JIT 内核执行。

### Q2：为什么不直接用 NumPy，要做编译器链路？
**答**：NumPy 能算数值，但不提供可控 IR、图级优化和多后端执行语义。我的目标是构建可演进基础设施：`IR -> pass -> codegen -> runtime`，这样才能做 fusion、memory planning、ABI 约束和训练期 kernel 化。

### Q3：广播梯度如何正确回传？
**答**：前向按 NumPy 广播规则，反向通过 dispatcher 中集中管理的 `_sum_to_shape/_expand_reduce_grad` 把梯度沿扩展维度做归约，确保梯度 shape 与原操作数一致；`sum/mean` 则通过 axis/keepdims 恢复或扩展后再广播。所有梯度规则在 `dispatcher.py` 的 `BINARY_GRAD/UNARY_GRAD` 中统一注册，消除了旧架构中每个算子内联闭包求导的冗余。

### Q4：你实现 backward 图时怎么做的？
**答**：对 forward graph 逆拓扑遍历，按算子定义局部梯度规则并做梯度累加；对广播场景在边上插入 reduce/broadcast 节点。最终把“loss 对每个参数”的梯度图分别编译为 JIT 模块。

### Q5：训练期 JIT 怎么避免每步重复编译？
**答**：采用 shape 签名缓存。首次遇到某输入 shape 才触发 trace+compile，后续同 shape 直接复用 compiled module；新 shape 才新增缓存项。

### Q6：为什么 optimizer 也要 kernel 化？
**答**：如果只编译前向/反向，参数更新仍在 Python 循环里，训练 step 仍有解释器开销。把 SGD/Adam update kernel 化后，完整训练闭环都可下沉到 JIT 执行路径。

### Q7：你做了哪些性能优化？
**答**：三层优化。图层：fusion + DCE。内核层：tiling + OpenMP + AVX。内存层：临时 buffer 生命周期复用，降低峰值内存与分配开销。

### Q8：如何保证运行安全性和可调试性？
**答**：通过 `TensorDesc` 严格 ABI 检查输入输出的 `shape/rank/stride/numel`，并返回结构化状态码；Python 侧统一解码错误，能快速定位是输入问题还是内核问题。

### Q9：惰性求值是怎么实现的？
**答**：新增 `LazyTensor`，算子只构建 thunk，不立即执行；在 `eval()/data/backward` 时才物化。并通过 `LazyJITTrainer` 把惰性构图与自动编译缓存统一起来。

### Q10：和工业级框架相比还差什么？
**答**：目前是可运行原型，差距主要在动态 shape 策略完整性、算子覆盖广度、调度与 cost model、profiling 工具链和分布式训练能力。但核心架构方向和关键技术路径已经打通，且经过 dispatcher 统一调度层重构后，架构的可维护性和扩展性已大幅提升（新增算子只需注册 eager kernel + 梯度规则）。

### Q11：你如何证明静态图不是“概念实现”，而是真能训练？
**答**：我提供了 `train_static_graph_linear_demo.py`，前端只用 `StaticGraph/SymbolicTensor` 构图，训练循环中调用编译后的 forward/backward/update kernels，最终 `final_mse=0.000398`，说明纯静态图训练闭环已可运行。

面试官问“为什么要用这个技术？没用之前卡在哪里？”这是非常经典的 **Trade-off（技术折衷）与痛点驱动** 的考察点。回答这类问题，你需要采用 **“痛点/瓶颈 (Before) -> 技术引入 (Action) -> 收益分析 (After)”** 的结构。

针对你的《Tensor CPU AI Compiler》项目，我为你梳理了**4个最核心技术点**的标准面试回答话术：

### 1. 为什么要引入 JIT 编译（C++ CodeGen）和计算图（Graph IR）？而不是直接用 Python/NumPy 算？
*   **没用之前卡在哪里（瓶颈）：**
    *   如果完全依赖纯 Python 的 Eager 模式执行，每次调用基础算子（如 add, mul）都要跨越 Python 和 C++ 底层（类似 PyTorch C++ extension或独立NumPy运算）。
    *   这会带来巨大的 **Python 解释器开销（Interpreter Overhead）**，尤其是遇到很多碎算子时，系统完全被调度开销卡住（CPU-bound在解释器上）。此外，Eager 模式每执行一次都要申请一块临时内存（Memory Allocation），内存抖动极大会导致严重的性能衰减。
*   **如何回复：**
    *   “为了解决上述痛点，我引入了 **Lazy/Static 构图与 JIT 后端生成技术**。程序先在前端构建 Graph IR，不执行具体数值；等到图确定后，直接通过 CodeGen 生成一整段 C++ 代码，通过 clang++ 动态编译成 `.so`，再通过 ctypes 传参一次性执行完毕。这样彻底 bypass 了 Python 解释器的调度开销，并且通过全局图视野，为后续的算子融合和内存复用打下了底层基础。”

### 2. 为什么要实现算子融合（Operator Fusion，如 MatMul + Add + ReLU）？
*   **没用之前卡在哪里（瓶颈）：**
    *   现代硬件的瓶颈往往在**访存带宽（Memory Wall）**而不是计算力。
    *   在未融合时，计算 `MatMul` 要从主存读取输入并将结果写回内存；接着计算 `Add`（加 Bias）又要再次读写内存；最后算 `ReLU` 还要再读写一次。每一次独立算子的执行都在浪费宝贵的总线带宽，导致程序的运行效率变成 **Memory-bound（访存密集型）**。
*   **如何回复：**
    *   “为了突破访存瓶颈，我在 Middle-end 实现了一个 **Pattern Matching Pass，将相邻的 MatMul、Add 和 ReLU 融合为一个大算子（FUSED_MATMUL_BIAS_RELU）**。这就意味着，我在 C++ Kernel 生成时，能在寄存器（Registers）或 L1 缓存级别直接完成加偏置和激活操作，再写回主存。这大幅减少了中间结果的内存 I/O 次数，使得网络在推理甚至反向传播时，性能获得了成倍提升。”

### 3. 矩阵乘（MatMul）为什么要用 AVX-2/AVX-512 加 OpenMP？不用不行吗？
*   **没用之前卡在哪里（瓶颈）：**
    *   最开始的原型我用的是朴素的三层 for 循环（Naive MatMul）。测试时发现，CPU 的占用率只有单核的 100%，而且连单核的峰值算力也发挥不出来。
    *   因为标量循环（Scalar Loop）一次只能处理一个浮点数，完全浪费了现代 CPU 强大的向量化计算单元（SIMD）和多核并行能力，项目在计算大规模网络（如 Attention 或 MLP）时处于极度的 **Compute-bound（计算力受限）**。
*   **如何回复：**
    *   “为了榨干 CPU 的硬件性能，我在 HPC 矩阵分块（BM/BN/BK）的基础上，引入了内联汇编/Intrinsics（`_mm512_*`, `_mm256_*` 等带 `fmadd` 乘加指令）实现向量化计算，一次能处理 8个或 16个 float32；并在外层并行维度加上了 `#pragma omp parallel for`。引入这些指令集和并行技术后，CPU 核心被全部打满，相比朴素实现，乘法性能提升了数十到上百倍，这也是我做底层框架的核心追求。”

### 4. 为什么要引入内存规划器（Memory Planner / 临时 Buffer 复用）？
*   **没用之前卡在哪里（瓶颈）：**
    *   在前向和反向求导时，计算图会产生大量中间节点（Activation/Gradient Tensor）。如果顺着算下来，每个中间 Tensor 都在 C++ Runtime 里去 `new` 内存，峰值内存（Peak Memory）会爆炸，甚至在 batch_size 稍微增大时直接 OOM（内存溢出）。
    *   并且频繁的 `malloc/free` 也会产生内存碎片，破坏 CPU Cache 的局部性（Cache locality）。
*   **如何回复：**
    *   “在图编译阶段（Middle-end），因为我们已经拿到了完整的算子拓扑排序和每个节点的生存期（Use-count/Lifetime），我就顺手实现了一个基础的 Memory Planner。对于已经出度为 0（生命周期结束）的中间节点缓冲，我让后面的节点直接无锁复用这块同等大小（同 numel）的内存。这样不但在运行时做到了 **Zero-allocation（零额外开销）**，而且大幅降低了训练的峰值内存开销。”

> “当时做这个项目的时候，一开始我只写了纯 eager 前端，跑那个 MLP 网络，结果发现慢得离谱。我用 profile 工具一查（或者粗略估算），发现时间根本没花在矩阵计算上，全是在 Python 层 `malloc` 和解释器调度。**因为卡在这里**，我才决定翻新架构，写了一套基于 Graph 的 IR 和 C++ CodeGen（顺势抛出你在架构上的重构）。生成 C++ 后速度提上来了，但我发现依然跑不过现成的库，分析后发现受限于访存和未被利用的 SIMD 单元。**为了去啃这块硬骨头**，我又引入了算子融合和 AVX512……”
