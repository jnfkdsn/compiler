我遇到了如何处理动态分支的问题。通过调研，我了解到了工业界像 TorchScript 的 AST 解析路线和 TorchDynamo 的 Bytecode 路线；由于精力有限，我的项目采用了轻量级的 Tracing 方案，但也实现了基于 Shape 的 Cache……” 这种回答能让你显得技术视野极大。

# 问题

**1. 你这个项目到底是什么定位？**
答：  
这是一个研究/教学导向的 CPU tensor compiler prototype，重点不是做一个完整工业框架，而是把 `Tensor API -> tracing -> Graph IR -> graph optimization -> C++ codegen -> 本地 JIT runtime` 这条链路完整打通。  
它现在已经具备真实系统的闭环，但 stable path 是刻意收窄的，比如只支持单输出，默认不支持 control flow lowering。

**2. 这个项目最核心的主路径是什么？**
答：  
主路径是 `Tensor -> TraceContext -> Graph -> optimize_graph -> CppCodegen -> JITEngine/runtime`。  
前端 `Tensor` 在 eager 模式下直接做 NumPy 计算，在 tracing 打开时会同时把操作记录为 `Graph` 节点；中端在 `Graph` 上跑常见优化；后端把图 lower 成 C++ kernel，并由 runtime 本地编译和执行。

**3. 为什么 Tensor 可以同时支持 eager 和 tracing？**
答：  
因为前端算子入口统一走 dispatch。  
同一个算子在 eager 模式下直接调用 NumPy 做数值计算，而在 tracing 开启时，会额外通过 tracer 往 `Graph` 中追加节点。  
这样做的好处是用户 API 保持一致，也方便用 eager 结果做开发期验证。

**4. 为什么要设计 Graph IR，而不是直接从 Tensor 表达式生成 C++？**
答：  
因为优化和后端生成都需要一个中间层。  
如果没有 `Graph IR`，就很难系统地做常量折叠、公共子表达式消除、融合、死代码消除，也不利于后端统一做 shape 推导、workspace 规划和 lowering。  
所以 `Graph` 是前端和后端之间的契约层。

**5. 你的 Graph 里除了依赖关系，还存了什么？为什么？**
答：  
除了输入输出关系，我还存了 `shape`、`dtype`、`rank`、`numel`、`strides` 和 `attrs`。  
这些元信息后面会被 shape inference、pass、memory planner 和 codegen 使用。  
比如 runtime 侧需要 rank 和 shape 做输入校验，codegen 侧需要 shape 和 attrs 生成具体 kernel 逻辑。

**6. 当前 pipeline 做了哪些优化？为什么先做这些？**
答：  
当前 stable pipeline 主要做四类优化：constant folding、CSE、fusion 和 DCE。  
这是一个比较轻量但性价比高的组合，因为它们对实现复杂度要求相对可控，但对生成代码质量和运行性能已经有明显帮助。  
我优先做这几类，是因为它们很适合作为 Graph IR 的第一批 canonical passes。

**7. 为什么 fusion 重要？**
答：  
fusion 可以减少中间张量 materialization，降低内存访问开销，也能减少 kernel 边界。  
在这个项目里，典型的例子是 `MatMul + Bias + ReLU`。  
如果不做融合，中间结果需要单独写回再读出；融合后可以在一个 kernel 里完成，更接近真实编译器会做的优化方向。

**8. 你的 codegen 是怎么组织的？**
答：  
当前 stable backend 已经从一个大类拆成了几个职责层：shape solver、memory planner、op lowering 和 C++ emitter。  
`CppCodegen` 本身更像 orchestration，它把图按拓扑顺序走一遍，先确定符号 shape 和存储规划，再对每个 op 做 lowering，最后发射成完整 C++ kernel 和相关元数据。

**9. 你为什么现在还是字符串生成 C++，而不是完整 AST？**
答：  
这是一个明显的工程取舍。  
用字符串发射能更快把主链路打通，尤其是在原型阶段成本更低；缺点是可维护性、重写和复杂控制逻辑支持都会比较差。  
所以我把它看成当前版本的实现手段，而不是最终形态，后续如果 layered backend 成熟，可以逐步往更结构化的 IR/AST lowering 演进。

**10. runtime 这一层做了什么？**
答：  
runtime 不只是“把代码编译一下”。  
它负责把生成的 C++ 写入临时文件、调用本地编译器生成动态库、通过 `ctypes` 加载符号，并在 `JITModule.run()` 中完成输入校验、输出分配、workspace 管理和 ABI 调用。  
也就是说，它承担了编译调度和执行时安全边界两部分职责。

**11. 你为什么要在 runtime 做 rank/shape guard？**
答：  
因为 native kernel 一旦假设了错误输入，最坏情况不是报错，而是 silent wrong answer。  
所以 runtime 先在 Python 侧做 rank 校验；如果是 HPC-specialized 2D matmul，还会做 exact traced shape guard，避免把静态特化 kernel 错用于另一组 shape。  
这是我在 correctness 上比较重视的一层保护。

**12. 你提到 HPC matmul 需要 exact traced shapes，为什么？**
答：  
因为这个路径会把 trace 时的部分尺寸直接特化进 kernel 模板。  
如果之后拿不同 shape 去复用同一个 kernel，输出 shape 可能表面上还能对，但数值会错。  
所以 stable path 里我选择保守策略：HPC 2D matmul 必须匹配 exact traced shape，这样先保证 correctness。

**13. 你怎么验证优化没有破坏正确性？**
答：  
我主要做了三层验证。  
第一层是单元/集成测试，比如 JIT correctness、fusion correctness、symbolic shape reuse、training correctness。  
第二层是 runtime guard，比如 rank guard 和 exact shape guard。  
第三层是 benchmark correctness guard，每个 benchmark case 在计时前都先和 NumPy reference 对齐，并记录 `max_abs_err`。

**14. benchmark 现在主要解决什么问题？**
答：  
它现在主要有三个作用：性能 smoke、回归检测、优化前后对照实验。  
我扩展过 benchmark，让它不只是跑几个 case，而是支持更多 workload、`HPC on/off`、`passes on/off`、`memory planner on/off` 的实验对照，并把 compile latency、median、p95 和 correctness guard 一起纳入报告。  
它现在适合做趋势判断和回归检测，但还不是完整 HPC 级实验体系。

**15. 这个项目最明显的限制是什么？**
答：  
最明确的限制有三个。  
第一，stable path 只支持单输出。  
第二，默认不支持 control flow lowering。  
第三，runtime 目前还是 `ctypes`-based，没有完全下沉到 C++ runtime library。  
此外，experimental 里的 layered codegen、control flow、pybind runtime 还没有完全并入主链。

**16. 为什么不直接把 experimental 的功能并入 stable？**
答：  
因为那样会让项目重新变乱。  
我现在更倾向于只保留一条稳定主线，然后把 experimental 的想法逐项晋升，而不是并行保留两三套体系。  
尤其是 control flow 和 layered backend，应该被吸收到同一套核心 IR，而不是长期和 stable path 平行存在。

**17. 你为什么还保留 Python 做前中端？**
答：  
因为前中端最重要的是迭代速度、表达力和调试效率，Python 在这方面非常合适。  
IR 构建、pass、shape inference、训练图拼接这些逻辑，用 Python 写能大幅降低原型成本。  
但我也很明确，runtime、allocator、binding、线程执行这些部分长期应该往 C++ 或 Rust 下沉。

**18. 如果继续做下去，你下一步最想改什么？**
答：  
我会按三个方向推进。  
第一，继续把 stable backend 的抽象拆清楚，避免逻辑重新堆回大类。  
第二，建立更严格的性能闭环，包括更稳定的 benchmark 和后续 perf counter。  
第三，把 runtime 下沉到 C++，让 Python 主要负责前端、编译调度和测试验证。

**19. 这个项目最值得拿来面试讲的点是什么？**
答：  
最值得讲的是它不是停在 IR 或 pass，而是完整打通了 end-to-end compiler/runtime loop。  
很多项目只做到 tracing 或只做到 codegen demo，但这个项目已经能把图编译成 native kernel，并且支撑 graph-mode training 和 benchmark/CI。  
这个“闭环感”是它最有讨论价值的地方。

**20. 如果面试官问：它和 TVM / XLA / PyTorch Dynamo 比有什么区别？**
答：  
我不会直接拿它做正面对标。  
更准确的说法是：这个项目借鉴了编译器系统里的典型分层和优化思想，但目标是构建一个可控、可解释、可端到端实验的 CPU tensor compiler prototype，而不是去覆盖 TVM/XLA 这类工业系统的广度和成熟度。  
它的价值在于系统思路和完整闭环，而不是生态和产品级能力。

**2. 讲一个具体例子：`relu(x @ w + b)`**
这题最好按数据流讲。

假设用户写的是 `relu(x @ w + b)`。

这个图里的每个节点不仅有依赖关系，还有 shape、dtype、rank、strides 等信息。  
接着 `optimize_graph` 会在图上做优化，如果识别到 `MatMul + Bias + ReLU` 这种模式，就可能融合成一个更大的算子，减少中间结果写回。

最后 `JITEngine` 会把这段 C++ 编译成本地动态库，`JITModule.run()` 在执行时会先做输入 rank/shape 校验、分配输出和 workspace，然后通过 ABI 调 native kernel。

**3. 讲 correctness guard**
这题要讲出“你不是只顾着让它跑快，而是知道编译器最怕 silent wrong answer”。

你可以这样答：

这个项目里我比较重视 correctness guard，主要做了三层。  
第一层是 runtime 输入校验，在 [runtime.py] 里会先检查输入个数、rank 是否匹配 traced graph，避免错误输入直接进入 native kernel。

第三层是 benchmark correctness guard。现在 benchmark 在计时前会先拿一组输入和 NumPy reference 对齐，并记录 `max_abs_err`，这样优化实验不会只看“更快了”，还会同时验证“没算错”。

这题的关键不是罗列 guard，而是讲出背后的原则：  
**在编译器和 runtime 里，错误结果通常比显式报错更危险，所以 guard 是必要的。**

> 第二个是项目演进过程中主路径变得不够清晰。随着 tracing、codegen、runtime 和一些 experimental 功能同时发展，代码开始出现多条并行路线。后来我做了一次收口，明确 stable path，把实验能力和稳定能力分开，同时把 codegen 继续按职责拆层。这样后面再做性能和 runtime 演进时，系统边界会更清楚。  

**回答时要避免**
- 不要说“最大的难点是时间不够”
- 不要说“就是 bug 很多，我慢慢修”
- 不要泛泛说“性能和正确性很难平衡”

# 反向传播与训练路径常见追问

**1. 这个项目里的反向传播是怎么实现的？**
答：
这个项目里有两条反向路径。
第一条是 eager 路径，在 [tensor.py](/home/jnfkdsn/tensor/tensor_cpu/tensor.py) 里，每个 `Tensor` 在前向时会记录前驱 `_prev` 和局部 `_backward` 闭包，调用 `backward()` 时按逆拓扑顺序执行，把梯度逐层传回去。
第二条是 graph-mode 路径，在 [train_jit.py](/home/jnfkdsn/tensor/tensor_cpu/autodiff/train_jit.py) 里，会基于 forward `Graph` 再构造一张 backward graph，然后交给同一套 `JITEngine` 编译。

**2. eager backward 和 graph-mode backward 的关系是什么？**
答：
它们不是两套完全独立的梯度数学实现，而是共享同一套 VJP 规则。
[vjp.py](/home/jnfkdsn/tensor/tensor_cpu/autodiff/vjp.py) 里每个算子同时定义了 eager 版和 graph 版梯度规则。
eager 路径用 NumPy 直接算梯度，graph-mode 路径则生成对应的梯度图节点。
这样可以避免梯度逻辑在多个地方重复实现。

**3. 什么是 VJP registry，为什么它重要？**
答：
VJP registry 可以理解成这个项目里反向传播的单一真源。
每个可微算子在 [vjp.py](/home/jnfkdsn/tensor/tensor_cpu/autodiff/vjp.py) 里都有统一注册项，里面同时包含 eager 和 graph 两种梯度定义。
它的重要性在于：前向加新算子后，不需要分别去改 eager backward、graph backward 和训练编译三套逻辑，而是只需要在一个地方补齐规则。

**4. eager 路径里的 backward 是怎么执行的？**
答：
eager 前向阶段，每个需要梯度的输出张量都会保存局部 `_backward` 函数和前驱 `_prev`。
调用 `backward()` 时，会先从当前张量出发做一次拓扑排序，再按逆拓扑顺序执行这些局部 `_backward`，把梯度累计到输入张量上。
这本质上是一个微型的反向自动微分执行器。

**5. graph-mode backward 是怎么从前向图生成的？**
答：
在 [train_jit.py](/home/jnfkdsn/tensor/tensor_cpu/autodiff/train_jit.py) 里，`build_backward_graph` 会先克隆 forward graph，然后从 loss 节点开始放一个梯度 seed，也就是标量 loss 对自身的梯度 1。
之后按 forward 图的逆拓扑顺序遍历节点，对每个节点调用 VJP rule，把输出梯度分解成各输入的梯度图节点，并把它们累加起来。
最后对每个目标参数或输入生成一张梯度图并标记输出。

**6. 为什么 graph-mode backward 里要求 loss 是标量？**
答：
当前实现里 backward builder 期望反向传播的起点是标量 loss，这样 seed gradient 可以自然设成 1。
如果输出不是标量，就需要更一般的 Jacobian 或外部传入 `grad_output` 语义，复杂度会明显上升。
所以 stable path 里我先把 backward builder 收窄到标量 loss。

**7. 训练编译链路具体做了什么？**
答：
训练链路不是只编译一个 forward。
在 [train_jit.py](/home/jnfkdsn/tensor/tensor_cpu/autodiff/train_jit.py) 里，`compile_training_step` 会先编译 forward loss graph，再为每个目标参数构造并编译 backward graph。
如果是优化器更新，还会额外编译 SGD 或 Adam update kernel。
所以训练阶段实际上是 loss、grad 和 update 三部分一起组成闭环。

**8. JITTrainer 做的事情和 compile_training_step 有什么区别？**
答：
`compile_training_step` 更偏底层，它直接接受 forward graph 和参数输入 id，返回 loss module 和 grad modules。
[jit.py](/home/jnfkdsn/tensor/tensor_cpu/nn/jit.py) 里的 `JITTrainer` 更偏上层封装，它会先 trace 模型和 loss，再调用 `compile_training_step`，然后把优化器更新和参数回写也串起来。
也就是说，一个偏底层编译原语，一个偏训练工作流封装。

**9. 为什么不直接沿用 eager backward，而要再做 graph-mode backward？**
答：
因为 eager backward 更适合解释执行和开发验证，但 graph-mode backward 的优势在于可以把梯度计算也变成图，再交给同一套后端做优化和本地编译。
这样 loss 和 grad 都能走统一 JIT 路径，更接近完整训练系统的形态。
从系统设计上看，这是把自动微分真正纳入编译链，而不是停留在 Python 解释执行层。

**10. 反向传播这部分目前的局限是什么？**
答：
局限主要有三个。
第一，backward builder 当前要求 scalar loss。
第二，stable path 里还没有把 control flow 和更复杂动态图场景纳入 graph-mode backward。
第三，虽然 eager 和 graph 共用 VJP 规则，但 runtime 仍然是 `ctypes`-based，训练执行链的底层运行时还不够工程化。
所以这部分已经有系统雏形，但还不是工业级训练编译器。

这个回答能体现你有架构收敛意识，而不是只会堆功能。

# 缓存管理和多线程
**一、缓存管理怎么实现**

当前主要有 **4 层缓存/复用**。

**1. codegen 阶段的临时 buffer 复用**
这是最接近“编译器内存规划”的部分，在 [memory_planner.py](/home/jnfkdsn/tensor/tensor_cpu/backend/memory_planner.py)。

它做的事情是：
- 先统计每个节点结果还会被后续用几次，`_compute_use_count(...)`
- 给中间节点分配临时存储，`_assign_node_storage(...)`
- 当某个中间结果最后一次被消费后，把它占用的 slot 放回空闲池，`_release_consumed_storage(...)`
它的复用策略比较简单直接：
- 用 `free_slots` 维护空闲 slot
- 按 **符号 shape** 分桶
- 后续遇到相同 shape 的临时结果，就优先复用旧 slot

所以这层不是“运行时 cache”，而是：
**编译时根据图生命周期做 workspace slot 复用**。
**2. runtime 阶段的 workspace arena 复用**
这层在 [runtime.py](/home/jnfkdsn/tensor/tensor_cpu/runtime.py) 的 `JITModule.run()` 里。

流程是：
- codegen 会生成 `workspace_slots` 的符号信息
- runtime 每次 `run()` 时根据实际输入 shape 算出需要的 workspace 大小
- `JITModule` 内部有一个 `_workspace_buf`
- 如果旧 buffer 足够大，就直接复用
- 不够大才重新分配
所以它本质上是：
**每个 compiled module 自己维护一个可增长的 workspace arena**
优点：
- 避免每次执行都重新 malloc 临时空间
- 对重复运行同一个 module 很有帮助
局限：
- 比较朴素，只是“够大就复用”
- 还没有更复杂的 allocator / alignment / pooling 策略
**3. eager JIT matmul 的 shape-based module cache**
这层在 [jit_matmul.py](/home/jnfkdsn/tensor/tensor_cpu/jit_matmul.py)。
如果你启用了 JIT-backed eager matmul，它会维护：
- `_JIT_MATMUL_CACHE: dict[(m, k, n, use_hpc), module]`
也就是说，同一个 2D matmul shape 再次出现时，不会重新 trace + codegen + 编译，而是直接拿缓存好的 `JITModule`。
这是一个非常典型的：
**按 shape 和配置做 kernel/module cache**
适合回答面试里的“JIT 编译后的结果有没有缓存”。
**4. 训练路径的 compiled kernel cache**
这层在 [jit.py](/home/jnfkdsn/tensor/tensor_cpu/nn/jit.py)。
有两种情况：
- `JITTrainer`
  - 第一次 `step()` 时 trace + compile
  - 之后同 shape 训练步骤复用同一套 compiled modules
- `LazyJITTrainer`
  - 维护 `_compile_cache`
  - 命中后直接复用 compiled entry
这层的本质是：
**训练工作流级别的 compile cache**
也就是不是缓存单个 op，而是缓存：
- forward loss kernel
- gradient kernel
- update kernel
顺便提醒你一个现实点：
这层 cache 目前更像“原型级缓存”，还不是成熟的通用 compile cache 系统。
**二、多线程部分怎么实现**

当前多线程并不是“整个 runtime 都线程池化”，而是比较集中地体现在 **HPC matmul kernel** 上。

主要在 [matmul_lowering.py](/home/jnfkdsn/tensor/tensor_cpu/backend/matmul_lowering.py) 的 `_emit_hpc_matmul(...)`。

它生成的 C++ 里会包含：

- `#pragma omp parallel for schedule(static)`
- `#pragma omp simd`

所以当前的多线程策略基本是：

- 对外层 block 循环做 OpenMP 并行
- 对内层向量循环做 SIMD 提示
- 再配合 AVX512 / AVX2 intrinsic 分支

这说明目前项目的“多线程”主要不是 runtime scheduler，而是：
**在特化 matmul kernel 里用 OpenMP 做 loop-level parallelism**

**现在还没有的部分**
当前没有完整做起来的多线程能力包括：

- 通用 thread pool
- runtime 统一调度多 op 并行
- task graph 并行执行
- NUMA 策略
- 更复杂的线程绑定策略
- allocator 和线程策略联动

所以别把它讲成“整个编译/runtime 系统都支持多线程调度”。  
更准确的说法是：

**当前多线程主要落在 HPC matmul kernel 这一条性能关键路径上。**


**四、如果你要一句话总结**

你可以这样说：

> 当前项目里的缓存管理主要分成四层：编译期的 workspace slot 复用、运行期 `JITModule` 内部的 workspace arena 复用、eager matmul 的 shape-based JIT module cache，以及训练路径对 compiled kernels 的复用。多线程目前主要集中在 HPC matmul kernel 上，通过 OpenMP 和 SIMD 做循环级并行，runtime 本身还没有演进成统一的线程调度器。

**五、如果面试官继续追问“局限是什么”**
你可以接一句：

> 这套缓存和并行策略现在已经能支撑原型和 benchmark，但还比较局部。缓存层面还没有统一的 compile cache / eviction / persistent cache 设计，多线程层面也主要集中在 matmul 特化 kernel，而不是整个 runtime 的通用并行执行模型。
