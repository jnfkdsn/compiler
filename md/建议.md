经历了上一轮拔除全局变量（引入 dispatcher.py）和剥离算子硬编码的重构后，目前这套架构在**前端追踪（Tracing）模型上已经具备了非常清晰的雏形**。但是，如果用工业级编译器（如 PyTorch Inductor, TVM, JAX）的标准来看，项目里目前仍然存在几个 **“设计断层”** 和 **“值得大改的痛点”**，主要集中在以下四个方面：

### 1. 最大的不合理：有两套互不相通的 Autograd 规则，维护成本翻倍
*   **现状**：
    *   在 dispatcher.py 中，你维护了 `BINARY_GRAD` 和 `UNARY_GRAD` 字典，里面写的是基于 NumPy 的 Eager 梯度运算（比如 `g * r.T`）。这是为了服务 `Tensor.backward()` 动态执行的。
    *   但在 train_jit.py 里的 `build_backward_graph` 函数，你又写了一个庞大的 `for n in reversed(ordered):` 循环，**把加减乘除所有的求导数学公式用图节点 `_add_binary()` 重新硬塞了一遍**。
*   **为什么不合理**：一旦未来引入新算子（比如 Conv2D 或 LayerNorm），开发者不仅要在 dispatcher 写一遍 eager 反向传播，还要在 train_jit 建图代码里再写一遍符号求导。一旦写错，两边行为就不一致了。
*   **改进思路（统一 VJP）**：参考 JAX 的设计，应该引入统一的 **VJP (Vector-Jacobian Product)** 抽象。反向传播规则应该只定义一次（基于高层 `Tensor` API）。不管是 Eager 执行还是 Graph Tracing，统统调用这一套高阶规则。既然 Tracing 时一切行为都会被录制成计算图，理论上 `backward` 图生成可以直接复用 eager 下的数学公式规则。

### 2. 严重的上下游脱节：Frontend 和 Backend 能力不匹配
*   **现状**：上一轮我帮你把 shape_inference.py（前端）和 `Tensor` API 放宽了：现在前端**合法支持了 float64**，以及 **N-维张量（N-D）的 Transpose 和 Batch Matmul**。
*   **为什么不合理**：
    *   我刚才查阅了你的 `codegen.py` 的第 368 行和 482 行，发现生成 C++ 代码时依然是**暴力写死的 2D 拆包**：`m, k = a_node.shape`。
    *   如果你现在向模型输入一个 3D 张量进行 `transpose` 或 `matmul`（这在 NLP Attention 里太常见了），前端和推理规则会跑通，**然后在 CodeGen 阶段引发崩溃 (`ValueError`)**。此外，C++ 的类型也完全硬编码成了 `float`，遇到 `float64` 解析就会出错或丢失精度。
*   **改进思路**：CodeGen 必须引入多维循环展开能力（N-D Loop Nest）。它不该假定维度，而是生成 `for (int d0...); for(int d1...)` 这种动态嵌套代码，并引入 C++ Templates 机制替代写死的 `float`。

### 3. 动态 Shape 风暴：循环上界严重硬编码
*   **现状**：在 JIT 编译时（如 `nn/jit.py` 和 CodeGen），你的 C++ 循环条件写的是具体的数字：`cond=f"j < {n}"`。缓存的 key 严格等于 `(x_shape, y_shape)`。
*   **为什么不合理**：如果你的 Batch Size 是动态的（比如大模型推理，有时是 32，最后那个 batch 是 16），那么改变一次 shape，系统就要触发整个 `clang++` 后端的**重新编译**。这在某些 NLP 轮循中会导致灾难性的耗时（Re-compilation Explosion）。
*   **改进思路（Symbolic Shape）**：编译时生成的 C++ 代码应该从硬编码 `j < 32` 变成去读传入的尺寸：`j < input_0.shapes[0]`。只要张量的 **Rank (维度数)** 没有变，就不需要重新触发 C++ 编译。

### 4. 运行时内存系统太弱：还在 C++ 侧动态堆分配 
*   **现状**：你的 `Buffer Pool` 复用虽然设计了生命周期统计，但这发生在生成的代码或者运行时内部。
*   **改进思路（Arena Workspace）**：现代框架是不允许内核计算时再去 `malloc` 甚至寻找 free-slot 的。在 Python 中端 `Pass` 分析完毕后，应该直接算出整张图最大需要的瞬时峰值内存长度（即 Memory Planner Planner Offset 算法），然后在 Python runtime 里**一次性申请一大块连续的 Memory Arena (Blob)**，把它作为指针 `void* workspace` 传给 C++。C++ kernel 里原本用来存放 `node_5` 的临时变量，全部换成 `float* n5 = workspace + offset_n5;`。这样能在底层彻底做到真正的 “Zero-Allocation运行时”。

### 5. 目前softmax的实现极易溢出，假设输入是一维向量或在某个轴（axis）上进行 Softmax 操作，一般遵循以下三个步骤：
